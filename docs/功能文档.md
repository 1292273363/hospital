# 医院微信小程序 — 功能文档

本文档面向产品与测试，概括本工程的需求背景、功能模块、技术架构及主要测试要点。

---

## 一、需求分析

### 1.1 项目定位

面向医院场景的微信小程序示例：用户通过微信登录后，可维护**患者档案**、查看**线下专家看诊记录**与诊断报告、发起**在线问诊**（上传疤痕图片与留言），并在**个人中心**管理账号信息与头像。

### 1.2 用户角色与目标

| 角色 | 目标 |
|------|------|
| 患者（小程序端） | 安全登录、选择当前就诊患者、查看公告与快捷入口、在线问诊、维护个人信息与患者档案 |
| 系统（后端） | 校验身份、持久化业务数据、对接微信登录与可选手机号、提供静态资源访问 |

### 1.3 核心业务规则（已实现）

- **登录态**：使用 JWT，请求头 `Authorization: Bearer <token>`；过期返回 `code=401` 并引导重新登录。
- **患者档案**：每条档案归属当前登录用户（`uploader_user_id`）；支持列表、详情、新增、修改、删除；**当前患者**缓存在本地与 `App.globalData`，供在线问诊等使用。
- **看诊记录**：与 `patient_record_id` 绑定；在线问诊仅展示**医生级别为「专家」**的记录。
- **在线问诊**：必须先查看所选专家记录的诊断报告，再上传图片并留言；图片存于项目根目录 `upload/consult/`。
- **模拟登录**：`mock-login-enabled: true` 时使用固定 mock openid，避免每次登录生成新用户导致档案“丢失”。

### 1.4 非功能需求

- 开发环境可本地联调；生产需 HTTPS 与微信合法域名。
- 数据库脚本统一放在仓库根目录 `sql/` 下，便于版本管理与执行。

## 二、各功能模块介绍

### 2.1 小程序端页面与职责

| 页面路径 | 模块名称 | 功能说明 |
|----------|----------|----------|
| `pages/login/login` | 登录 | 协议勾选、微信一键登录（患者端）、医生登录入口 |
| `pages/doctor/login` | 医生登录 | 手机号 + 验证码登录（`role=doctor`） |
| `pages/doctor/index` | 医生端首页 | 医生登录后的工作台入口（当前为基础占位页） |
| `pages/index/index` | 首页 | 用户信息展示、快捷入口（含在线问诊跳转）、医院公告、退出 |
| `pages/center/index` | 中心（Tab） | 头像上传、当前患者展示、个人中心、患者档案入口、退出 |
| `pages/patient/index` | 患者档案 | 档案列表、切换当前患者、新增/修改/删除档案 |
| `pages/upload/index` | 患者表单 | 新增或编辑患者档案（由患者档案页带参数进入，非 Tab） |
| `pages/profile/edit` | 个人中心 | 修改昵称、手机号、头像 URL |
| `pages/consult/index` | 在线问诊 | 基于当前患者加载专家看诊记录、查看报告、上传疤痕图与留言、历史记录 |

**底部导航（TabBar）**：首页、中心（已移除独立「上传」Tab，患者信息统一走患者档案）。

**公共能力**：`utils/request.js` 统一携带 Token、处理 401 与通用错误。

### 2.2 后端模块划分

| 包/类 | 职责 |
|--------|------|
| `controller.AuthController` | 角色化验证码发送与登录（patient/doctor） |
| `controller.WxLoginController` | 微信登录、健康检查 |
| `controller.UserController` | 当前用户信息查询/更新、头像文件上传 |
| `controller.PatientController` | 患者档案 CRUD、按手机号聚合查询看诊信息 |
| `controller.OnlineConsultationController` | 可选专家看诊列表、报告、问诊提交、问诊历史 |
| `config.JwtAuthFilter` | `/api/*` 除白名单外 JWT 校验 |
| `config.WebConfig` | 跨域、`/upload/**` 静态映射到根目录 `upload/` |
| `service.impl.WxLoginServiceImpl` | code2session、用户创建/更新、手机号、JWT 签发 |
| `entity.*` + `*Mapper` | MyBatis-Plus 实体与数据访问 |

| 文件 | 说明 |
|------|------|
| `001_20260331_patient_record.sql` | 患者信息表 `patient_record` |
| `002_20260331_online_consultation.sql` | `visit_record`、`online_consultation` 等基础表 |

## 三、整体技术架构

### 3.1 逻辑架构

```
┌─────────────────┐     HTTPS/JSON      ┌──────────────────────────┐
│  微信小程序      │ ◄─────────────────► │  Spring Boot 3.2 后端     │
│  (原生 WXML/JS) │   multipart 上传    │  JWT 鉴权 + REST API       │
└─────────────────┘                     └────────────┬─────────────┘
                                                     │
                                                     ▼
                                            ┌────────────────┐
                                            │  MySQL 8.0     │
                                            └────────────────┘
```

### 3.2 技术栈

| 层次 | 技术 |
|------|------|
| 小程序 | 微信原生小程序、TabBar PNG 图标 |
| 后端 | Java 17、Spring Boot 3.2、Spring Web |
| 持久化 | MyBatis-Plus 3.5、MySQL |
| 安全 | JWT（jjwt 0.12）、Filter 白名单 |
| 外部依赖 | 微信 jscode2session、可选 getuserphonenumber |

### 3.3 接口一览（需登录除登录与健康检查）

**白名单（无需 Token）**

- `POST /api/wx/login` — 登录
- `POST /api/auth/send-code` — 发送验证码（`role=patient/doctor`）
- `POST /api/auth/login` — 角色化登录（`role=patient/doctor`）
- `GET /api/wx/health` — 健康检查

**用户**

- `GET /api/user/me` — 当前用户信息
- `PUT /api/user/me` — 更新资料
- `POST /api/user/avatar` — 上传头像（`multipart/form-data`，字段 `file`）

**患者档案**

- `GET /api/patient/list` — 我的患者档案列表
- `GET /api/patient/{id}` — 档案详情
- `POST /api/patient/upload` — 新增档案
- `PUT /api/patient/{id}` — 修改档案
- `DELETE /api/patient/{id}` — 删除档案
- `GET /api/patient/lookup-by-phone?phone=` — 按档案手机号查询关联看诊等（测试/管理用途）

**在线问诊**

- `GET /api/consult/eligible-visits?patientRecordId=` — 当前患者下专家看诊列表
- `GET /api/consult/report/{visitId}` — 诊断报告（校验权限与专家）
- `GET /api/consult/history?patientRecordId=` — 该患者相关问诊历史
- `POST /api/consult/submit` — 提交问诊（`visitId`、`message`、`file`）

统一响应体：`{ code, message, data }`；业务失败多为 `code=500`，未登录或 Token 失效为 `code=401`。

## 四、系统主要功能与测试

### 4.1 功能清单（建议测试矩阵）

| 编号 | 功能 | 前置条件 | 预期结果 |
|------|------|----------|----------|
| T1 | 微信登录 | 配置合法 appid/secret 或开启 mock | 返回 token 与 userInfo，本地存储成功 |
| T1-1 | 医生验证码登录 | 手机号可用，后端可访问 `/api/auth/*` | 验证码发送成功并可进入医生端首页 |
| T2 | Token 过期 | 篡改或等待过期 | 接口返回 401，跳转登录页 |
| T3 | 患者档案新增 | 已登录 | 列表出现新档案，可选为当前患者 |
| T4 | 患者档案修改/删除 | 已登录且档案属于自己 | 数据更新或删除；删除当前患者时需关注是否清空本地 currentPatient |
| T5 | 切换当前患者 | 多条档案 | 在线问诊拉取的看诊列表随患者变化 |
| T6 | 专家看诊数据 | `visit_record` 中 `doctor_level=专家` 且 `patient_record_id` 正确 | 问诊页可选记录 |
| T7 | 查看诊断报告 | 已选专家记录 | 展示报告内容 |
| T8 | 提交在线问诊 | 已查看报告、已选图片 | 记录写入 `online_consultation`，图片在 `upload/consult/` |
| T9 | 问诊历史 | 有提交记录 | 列表展示图片、留言、状态、专家回复字段 |
| T10 | 头像上传 | 已登录 | 文件在 `upload/`，用户头像 URL 可访问 |
| T11 | 按手机号查看诊 | 已登录且手机号对应本人档案 | `lookup-by-phone` 返回档案与看诊列表 |
| T12 | 模拟登录稳定性 | `mock-login-enabled: true` | 多次登录同一 wx_user，患者档案不丢 |

### 4.2 测试数据建议

- 执行 `sql/` 下建表脚本后，先创建患者档案（或走小程序新增），再执行 `005_20260401_visit_test_dataset.sql` 等，确保 `patient_record_id` 与业务一致。
- 同一手机号若存在多条 `patient_record`，看诊绑定以脚本中「按 id 降序取最新」为准，测试时注意选用档案与数据是否一致。
- 医生端测试号建议：`13800138000`（或任意 11 位手机号）。开发环境验证码通过医生登录页“获取验证码”弹窗直接查看。

### 4.3 已知限制与注意点

- 生产环境应关闭 mock 登录，使用真实微信 code；并缓存 access_token、加固密钥管理。
- 「专家回复」需后续扩展医生端接口与权限模型，当前表结构已预留字段。
- 首页部分快捷入口仍为占位（Toast「功能开发中」），仅在线问诊与个人中心相关流程完整度较高。
